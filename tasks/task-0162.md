## Research Findings — AI Memory Systems 2026

### The Memory Problem
LLMs are stateless by default — each session starts fresh. Memory systems solve this by persisting and retrieving what matters across sessions.

### Key Types of AI Memory

**1. Short-term / Context Memory**
- The LLM's context window (e.g., 128K tokens for GPT-4o)
- Conversation buffer (last N messages)
- Cheap, fast, but limited

**2. Long-term / Persistent Memory**
- Stores facts, preferences, history beyond the session
- Must be retrievable at query time
- Powers personalization

**3. Semantic / Vector Memory**
- Facts and documents stored as embeddings in a vector DB
- Retrieved by semantic similarity (not keyword)
- Powers RAG: "retrieve relevant knowledge → inject into prompt"

**4. Graph Memory**
- Relationships between entities stored as a knowledge graph
- Enables multi-hop reasoning ("who works with whom")
- Mem0 and others now combine this with vector for hybrid retrieval

### Leading Memory Systems

**Mem0** (open source)
- Self-improving memory layer for AI agents
- Multi-level: user-level, session-level, agent-level memory
- Hybrid retrieval: vector search + metadata filtering + optional graph
- Framework-agnostic, easy API
- `pip install mem0ai`

**Letta** (formerly MemGPT)
- Agent *runtime* — not just a library, a full server
- Agents manage their own memory hierarchy: context ↔ archival storage
- Organizes memory into: Subjects, Blocks, Messages
- REST API + Agent Development Environment (ADE)
- Emphasis: agents that self-edit their memory over time

**Zep** (still active 2026)
- Cloud + self-hosted options
- Strong enterprise features: user privacy, data retention policies
- Useful for production chatbots needing session memory

### RAG vs. Memory — What's the Difference?

| | RAG | Dedicated Memory |
|--|-----|-----------------|
| Purpose | Ground LLM in external knowledge | Persist user/session state |
| Data | Documents, wikis, databases | Preferences, facts, history |
| Retrieval | Semantic vector search | Both semantic + structured |
| Persistence | Usually static | Continuously updating |
| Use case | "What does our policy say?" | "What does this user prefer?" |

Most production agents combine both: RAG for domain knowledge + memory for personalization.

### How It Works (Simplified)

```
User Message → Embed → Retrieve from Vector DB
                           → Retrieve from Memory Store
                           → Combine with system prompt
                           → LLM generates response
                           → Store key facts back in memory
```
