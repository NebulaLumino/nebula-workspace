## Research Summary

### Why Agent Evaluation Is Hard
Evaluating AI text generation is already tricky — you need good judges, diverse test sets, and understand what "good" means. Evaluating AI agents is an order of magnitude harder: agents take multiple steps, use tools, produce artifacts, and succeed or fail in open-ended ways. Traditional LLM benchmarks don't cut it.

### The Evaluation Stack for AI Agents

**Benchmarks (Standardized Tests)**

1. **GAIA (General AI Assistants)**
   - 300+ real-world questions requiring web search, code execution, file manipulation
   - Tests multi-step reasoning + tool use
   - Human-verified answers
   - Current best: ~40% accuracy (plenty of room to improve)

2. **AgentBench**
   - Multi-domain agent evaluation: OS, database, knowledge graph, digital card games
   - Tests actual tool use in realistic environments
   - 8 different domains
   - Used to benchmark LangChain, AutoGen, and other frameworks

3. **WebArena**
   - Agents browse real websites (not simulated)
   - Tasks: find information, book things, navigate sites
   - Hard because real websites are complex and changing

4. **MiniWob++ / WebGym**
   - Tiny browser automation tasks
   - Foundation for web agent research
   - Easy to run at scale

5. **MINT-Bench**
   - Multi-image reasoning + tool use
   - Tests multimodal agents

6. **API-Bank**
   - Tool use evaluation specifically for API-calling agents
   - 73 tools, 264 test cases

**Evaluation Approaches**

**1. Sandbox Execution + Outcome Verification**
- Let agent complete task in sandbox (EC2, Docker)
- Check: Was the file created? Was the email sent? Did the code run?
- Pros: objective pass/fail
- Cons: only works for verifiable outcomes

**2. LLM-as-Judge**
- Use GPT-4 or Claude to evaluate agent output
- Give judge: task description, agent output, reference solution
- Pros: handles open-ended tasks
- Cons: judge bias, expensive

**3. Human Evaluation**
- Real humans rate agent outputs
- Still gold standard for subjective quality
- Cons: slow, expensive, hard to scale

**4. Reference Traces**
- Compare agent's trajectory to known-good trajectory
- Measures process (how it got there) not just outcome
- Good for debugging failures

**5. Self-Evaluation / Reflection**
- Agent evaluates its own output
- Cheap but unreliable (agents over-rate themselves)

### What to Measure in Agent Systems

| Dimension | Metrics |
|-----------|---------|
| **Task Success** | % tasks completed, pass/fail rate |
| **Efficiency** | Steps taken, time to completion, API calls |
| **Reliability** | Consistency across multiple runs |
| **Safety** | Rate of unsafe tool calls, jailbreak attempts |
| **Cost** | Total compute cost per task |
| **Error Recovery** | Does it recover from mistakes? |
| **Context Efficiency** | How well does it use context? |

### The AgentEval Framework (Microsoft)
- Custom benchmark creation tool
- Define task → define success criteria → run agent → get scored
- Used internally at Microsoft for agent development
- Could be adapted for building evaluation pipelines

### Continuous Evaluation / CI for AI
- Run agent against test suite on every code change
- "AI unit tests" — example inputs with expected outputs
- Regression testing: did agent quality degrade?
- LangSmith, Phoenix (Arize), Weights & Biases all offer agent tracing + evaluation

### Open Problems in Agent Evaluation
1. **Coverage**: How many scenarios to test? (impossible to exhaust)
2. **Distribution shift**: Agent succeeds on test, fails on new real-world inputs
3. **Emergent failures**: Rare edge cases only appear in production
4. **Process vs. outcome**: Good process can lead to bad outcome (and vice versa)
5. **Human preference**: Different humans prefer different agent behaviors
